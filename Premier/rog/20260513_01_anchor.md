クリップが画面外に吹き飛ぶバグ。

## 2. EG クリップへのアンカーポイント制御対応

### EG クリップのコンポーネント構造（確認済み）

エフェクトコントロール表示:
```
グラフィック（クリップ）
├── ベクトルモーション         ← クリップ全体の変換（シーケンス座標ピクセル）
│   ├── 位置: 960, 540
│   └── アンカーポイント: 960, 540
└── テキスト（EGレイヤー）
    └── モーション             ← EG内部レイヤーの変換（EG内座標ピクセル）
        ├── 位置: 960, 540
        └── アンカーポイント: 0, 540 など
```

CEP の `clip.components` 走査で:
- **「モーション」** は displayName で見つかる（テキストレイヤー内のもの）
- **「ベクトルモーション」** は displayName 検索では見つからない
  → 「グラフィック」コンポーネントの埋め込みプロパティとして存在している可能性

### ベクトルモーション検索の対応

「ベクトルモーション」という名前のコンポーネントが見つからない場合の fallback として、  
モーション以外のコンポーネントで 位置 + アンカーポイント プロパティを持つものを検索する実装を追加。

```javascript
// ① 名前で明示検索
if (cdn === 'Vector Motion' || cdn === 'ベクトルモーション') vectorMotionComp = comp;
// ② モーション以外のコンポーネントでAP+Positionを持つものを検索（ベクトルモーション fallback）
if (!vectorMotionComp && cdn !== 'Motion' && cdn !== 'モーション') {
    // comp.properties に アンカーポイント+位置 があれば vectorMotionComp として使う
}
```

**テスト結果: ベクトルモーション モードは正常動作確認。**

## 3. Hold Position スケール配列バグ（修正済み）

EGクリップで `scaleProp.getValue()` が `[100,100]`（配列）を返す。  
`[100,100] / 100 = NaN` → 位置補正が壊れる。

**修正:**
```javascript
var rawScale  = scaleProp.getValue();
var curScale  = (rawScale !== null && typeof rawScale === 'object') ? rawScale[0] : (rawScale || 100);
```

## 5. EG内部モード 4方向テスト結果と問題（2026-05-13）

### テスト条件

Hold Position OFF / ON 両方でテスト（EG内部 モード）。

### アンカーポイントの変化（Hold OFF）

| ボタン | xMult/yMult | 期待値 | ベクトルAP | テキストAP（結果） |
|---|---|---|---|---|
| 右中央 | 1.0, 0.5 | 1920, 540 | 960, 540 変化なし | **1920, 540** |
| 左中央 | 0.0, 0.5 | 0, 540 | 960, 540 変化なし | **0, 540** |
| 上中央 | 0.5, 0.0 | 960, 0 | 960, 540 変化なし | **960, 0** |
| 下中央 | 0.5, 1.0 | 960, 1080 | 960, 540 変化なし | **960, 1080** |

→ アンカーポイントの値自体は正しく設定されている（`seqW * xMult, seqH * yMult`）。  
→ ただし視覚的に「正常には動かない」（テキストレイヤーの座標系の問題）。

### Hold Position ON の問題（→ 修正済み）

- 位置（ベクトルモーション の位置ではなくテキストのモーション の位置）は維持される ✅
- しかしアンカーポイントが変化しない ⚠️（旧実装）

**原因:** EG内部レイヤーでは「位置を変更すると Premiere がアンカーを自動調整する」という一方向の連動がある。  
旧実装: anchor → position の順にセット → position セット時に Premiere がアンカーをリセット → アンカーが元に戻る。

**修正（実装済み）:** `_snapComponent` で Hold ON 時のセット順を逆にした。  
```
// 旧: anchor → position （position setValue で Premiere がアンカーをリセット）
// 新: position → anchor （anchor setValue は Premiere の自動調整を引き起こさない）
posProp.setValue([newPosX, newPosY], 1);   // 位置を先にセット
anchorProp.setValue([targetAX, targetAY], 1); // アンカーを後にセット（Premiere はここでは位置を弄らない）
```
この逆順により、最終的にアンカーが target 値に確定する（要テスト）。

## 6. EG内部モードの座標系（未解決）

### 現在の isPixelUnits 判定

```javascript
var isPixelUnits = (Math.abs(curAnchor[0]) > 2 || Math.abs(curAnchor[1]) > 2);
```

EG内部 モーション の初期アンカーが (0, 540) の場合:
- `Math.abs(540) > 2 = true` → ピクセル判定される
- `targetAX = newAX * seqW, targetAY = newAY * seqH`
- 例: 右中央 → (1920, 540) がセットされる

ただし EG内部のモーションが **中央原点座標系**（(0,0)=EG中心）を使っていた場合:
- 正しい右中央の値は (+960, 0)（EG幅の半分）
- 現在の実装では (1920, 540) をセットしてしまい、大きくずれる

### AS_dumpComponents 診断関数を追加

EGクリップ読み込み時にコンポーネント一覧と各プロパティ名・値を DevTools コンソールへ出力。  
DevTools（パネルを右クリック → 開発者ツール）で確認することで、  
EG内部モーションの正確な座標系を特定できる。

## 7. EG内部モードの制御対象を修正（2026-05-13）

### 問題の再整理

ユーザー指摘: 「EGのアンカーポイント＝プロパティパネルの整列と変形のアンカーポイント」

| パネル | 対象 | アンカーポイント値 | 座標系 |
|---|---|---|---|
| エフェクトコントロール > モーション | シーケンス全体 | 960, 540 | シーケンス空間ピクセル |
| プロパティ > 整列と変形 | EGレイヤー本体 | 215, 256.5 | シェイプ内部ピクセル |

→ これまでのコードは Effect Controls の「モーション」コンポーネント（シーケンス空間）を操作していた。  
　正しい対象は**プロパティパネルの「整列と変形」に対応するEGレイヤーコンポーネント**。

### 修正内容

**コンポーネント検索の追加:**  
`clip.components` をスキャンして `ベクトルモーション` でも `モーション` でもなく、  
`アンカーポイント` プロパティを持つコンポーネント（シェイプ01等）を `egLayerComp` として検出。

**座標計算の修正:**  
EGレイヤーのアンカーポイントはシェイプ内部ピクセル空間（0, 0）〜（W, H）。  
`_getEGLayerSize(comp)` でシェイプのW/Hを取得し、`targetAX = xMult * layerW` で計算。  
W/Hが取得できない場合は seqW/seqH にフォールバック（要調査）。

**Hold Position の順序:**  
position → anchor の順でセット（Premiere の自動調整対策、前回修正済み）。

## 8. EG内部モード W/H 取得の未解決問題（2026-05-13）

### 試みたアプローチ

**アプローチA: AS_getClipInfo でクリップ読み込み時に egLayerAnchor を取得**  
EGレイヤーコンポーネントのアンカー初期値を読み取り、「デフォルトは中央」と仮定して  
`W = anchor[0] * 2, H = anchor[1] * 2` でサイズ推定 → **失敗（EG: W/H不明 が出続ける）**

### 失敗の原因判明

実際のEGシェイプのプロパティパネル値（テスト時のスクショより）:
```
アンカーポイント: 0X, 540Y   ← X=0 なので ia[0] > 2 チェックが失敗する
位置: 616X, 379.5Y
W: 288,  H: 321
```

- コードの条件 `ia[0] > 2 && ia[1] > 2` → ia[0]=0 なので **false** → egLayerW=0 のまま
- アンカーが中央でなく左端(X=0)にあったため「中央の仮定」が崩れた
- 加えて Y=540 は H=321 より大きくシェイプ外の値 → 座標系がまだ不明

### EG プロパティパネルの座標系（未確定）

観測値:
| プロパティ | 値 |
|---|---|
| アンカーポイント | 0X, 540Y |
| 位置 | 616X, 379.5Y |
| W | 288 |
| H | 321 |

- Y=540 がシーケンス垂直中央（1080/2=540）に一致 → **シーケンス空間** の可能性
- Y=540 がシェイプ H=321 を超えている → **レイヤーローカル座標ではない** 可能性
- X=0 が構成上の左端（シーケンスまたはEGキャンバス左端）の可能性
- 座標系: シーケンス空間ピクセル（0,0=左上, 1920x1080）と思われるが未確定

### 影響

- `egLayerW/H` が 0 のまま → `_snapComponent` の isPixelUnits フォールバック:  
  `targetAX = xMult * seqW = xMult * 1920`  
  これは**シーケンス空間で正しい可能性**がある（座標系がシーケンス空間なら）
- 要実験: seqW/seqH フォールバックのままでもアンカーが意図した位置に来るか確認
